# Микрофон при запуске через Wine на macOS

## Симптомы

Клиент запускается через Wine (CrossOver, Whisky, PlayOnMac, «голый» wine из
Homebrew) на macOS. Список устройств ввода отображается нормально, устройство
выбирается, но звук не захватывается — индикатор микрофона не шевелится,
передаётся тишина.

Начиная с этой версии клиент сам определяет такую ситуацию: при первом запуске
под Wine на macOS показывается подсказка, а если микрофон в течение ~10 секунд
отдаёт чистую цифровую тишину, в лог выводится предупреждение со ссылкой на
этот документ.

## Причина

Начиная с macOS 10.14 (Mojave) доступ к микрофону выдаётся приложениям
персонально через систему разрешений TCC («Конфиденциальность и безопасность»).
Приложение обязано запросить разрешение, и macOS показывает диалог
«… хочет получить доступ к микрофону».

Wine **не пробрасывает этот запрос** из Windows-окружения: Windows-приложение
внутри Wine обращается к WASAPI, Wine транслирует это в CoreAudio, а macOS
молча отдаёт тишину, потому что разрешение никто не запросил и не получил.
Ошибки при этом нет — устройства перечисляются, поток «захватывается», просто
все сэмплы нулевые.

Разрешение выдаётся **не Windows-приложению, а тому macOS-приложению, которое
запускает Wine** («ответственному процессу»).

## Решение

### 1. Выдать разрешение приложению-лаунчеру

Откройте на Mac: **Системные настройки → Конфиденциальность и безопасность →
Микрофон** и включите переключатель у приложения, из которого запускается
Wine:

| Как запускаете | Кому выдать разрешение |
|---|---|
| Из Терминала (`wine mumble.exe`) | Terminal (или iTerm) |
| CrossOver | CrossOver |
| Whisky | Whisky |
| PlayOnMac | PlayOnMac |
| Wine Stable.app (пакет WineHQ) | Wine Stable |

После включения **полностью перезапустите** Wine (`wineserver -k`) и клиент.

### 2. Если приложения нет в списке

macOS добавляет приложение в список только после того, как оно *запросило*
доступ. Если Wine собран без ключа `NSMicrophoneUsageDescription` в
`Info.plist`, запрос не происходит и приложение в списке не появляется.

Запустите вспомогательный скрипт (на Mac, не внутри Wine):

```bash
./scripts/macos-fix-wine-mic.sh
```

Он найдёт бандл Wine, добавит в его `Info.plist` ключ
`NSMicrophoneUsageDescription`, переподпишет бандл ad-hoc подписью и сбросит
кэш разрешений — при следующем запуске macOS покажет диалог запроса доступа.

То же самое вручную:

```bash
# путь к бандлу может отличаться
APP="/Applications/Wine Stable.app"
/usr/libexec/PlistBuddy -c 'Add :NSMicrophoneUsageDescription string "Voice chat needs microphone access"' "$APP/Contents/Info.plist"
codesign --force --deep --sign - "$APP"
tccutil reset Microphone
```

### 3. Если разрешение выдано, а звука всё нет

- Сбросьте кэш разрешений и дайте macOS спросить заново:
  `tccutil reset Microphone` — затем перезапустите Wine и клиент.
- Проверьте, что микрофон вообще работает на Mac (Системные настройки → Звук →
  Вход).
- В настройках клиента (Настройки → Аудиовход) выберите устройство «Default» —
  Wine надёжнее всего пробрасывает системное устройство по умолчанию.
- Убедитесь, что используется современный Wine (8.0+): в старых версиях
  захват через winecoreaudio работал нестабильно.

## Проверка

Откройте «Настройки → Аудиовход» или мастер настройки аудио: индикатор уровня
должен реагировать на голос. Если клиент больше не пишет предупреждение о
тишине — захват работает.
