# Система обновлений mumble-tfar (Storm Voice)

Как клиент находит, скачивает и устанавливает обновления, и почему версия
Windows-установщика отличается от версии приложения.

## Как работает обновление

1. При запуске (и при ручной проверке) `src/mumble/VersionCheck.cpp` запрашивает
   последний релиз через GitHub API (`STORM_RELEASES_API` в
   `src/mumble/tfar/StormBranding.h`).
2. `tag_name` релиза (без ведущего `v`) сравнивается с `STORM_TFAR_VERSION`,
   зашитым в бинарник. Если релиз новее — пользователю предлагается обновиться.
3. Клиент скачивает ассет `mumble_client-*.exe` (bootstrapper; запасной вариант —
   `.msi`) во временную папку, пишет туда же helper-скрипт
   `storm-voice-update.cmd` и запускает его, после чего закрывается.
4. Скрипт ждёт пару секунд (чтобы клиент освободил файлы), запускает установщик
   в тихом режиме (`/passive /norestart` — только прогресс-бар, без мастера) и
   после завершения установки перезапускает клиент по прежнему пути.

Пользовательские настройки установщик не трогает: они лежат в профиле
пользователя (Qt/QSettings), а MSI ставит только файлы программы в
`%ProgramFiles%\Storm of The Galaxy\client`.

## Версия установщика ≠ версия приложения

Версия приложения — `STORM_TFAR_VERSION` из `StormBranding.h`; релизный workflow
(`.github/workflows/release-tfar.yml`) подставляет туда версию из тега перед
сборкой.

Версия MSI/EXE-установщика вычисляется в секции packaging
`src/mumble/CMakeLists.txt` из `STORM_TFAR_VERSION` **со сдвигом мажорной
компоненты на +1**: релиз `1.2.1` собирает установщик `2.2.1`, релиз `1.3.0` —
`2.3.0` и т.д.

Зачем сдвиг: релизы до v1.2.0 включительно штамповали каждый MSI версией Mumble
(`PROJECT_VERSION` = `1.7.0`, потому что build number в релизной сборке равен 0).
Из-за этого:

- Windows Installer не видел «обновления» (новая версия не больше старой) —
  major upgrade не срабатывал, и каждый релиз регистрировался **отдельной
  копией** в «Программы и компоненты»;
- файлы с той же файловой версией не заменялись — после «обновления» запускался
  старый `mumble.exe`, и клиент снова сообщал о старой версии;
- задвоенные продукты с общими путями и ключами реестра ломали друг другу
  файлы и ассоциации при удалении/repair.

Сдвиг мажора гарантирует, что любой новый установщик (`2.x.y`) строго больше
всех легаси-продуктов (`1.7.x`), поэтому:

- **MSI**: major upgrade (`UpgradeVersions = ThisAndOlder`,
  `RemoveExistingProductAfter = InstallInitialize` в
  `installer/MumbleInstall.cs`) находит по upgrade GUID и удаляет **все**
  установленные копии — и легаси `1.7.0`-дубликаты, и копию той же версии при
  переустановке, — а затем кладёт файлы заново;
- **EXE (Burn-bundle)**: версия bundle берётся из MSI, upgrade GUID общий —
  bundle новой версии заменяет регистрации старых bundle, а не встаёт рядом.

Upgrade GUID (`s_UpgradeGuid` в `installer/ClientInstaller.cs`) менять
**нельзя** — иначе новые установщики перестанут находить и вычищать старые
установки.

## Совместимость со старыми клиентами

Ничего дополнительного не требуется: уже установленные старые клиенты сравнивают
только тег релиза со своей `STORM_TFAR_VERSION`, скачивают свежий установщик и
запускают его. Свежий установщик сам вычищает все продукты-дубликаты и ставит
новую версию — переход бесшовный, даже если у пользователя накопилось несколько
сломанных копий.

## Чек-лист релиза

1. Убедиться, что тег имеет вид `vX.Y.Z` и `X.Y.Z` больше предыдущего релиза
   (сравнение покомпонентное).
2. Запушить тег — workflow сам подставит версию в `StormBranding.h`, соберёт
   клиент, MSI/EXE (версией `X+1.Y.Z`), portable ZIP и MSIX и опубликует релиз.
3. `STORM_TFAR_VERSION` в `StormBranding.h` в репозитории желательно поднять до
   той же версии (значение по умолчанию для локальных сборок).
