# 多群组收听：基线验收与产品映射

## 1. 基线验收（源码核对，1.5.901）

| 验收项 | 结论 | 证据 |
|--------|------|------|
| A 在频道1 PTT，B 监听1但身在2 → B 能听到 | **已满足** | `Server::processMsg` 对 `REGULAR_SPEECH` 调用 `getListenersForChannel` |
| B 在2 PTT → 仅 2 的成员 + 监听 2 的人听到 | **已满足** | 发送者路由仅用 `u->cChannel`，监听集合不影响发送目标 |
| 无 `Listen` 权限无法 add | **已满足** | `Messages.cpp` `hasPermission(..., ChanACL::Listen)` |
| 重连后持久化 | **已满足（可配置）** | DB `channel_listeners`；认证后 `loadChannelListenersOf` |
| 默认上限 | **默认无上限**（`-1`） | `Meta.cpp`：`iMaxListenersPerChannel/iMaxListenerProxiesPerUser = -1`；ini 示例注释值为 5/2 |

### 真实缺口（相对业务「多群组」心智）

1. **无独立「多群组」管理面板**；仅有频道右键「Listen To Channel」、快捷键、Listener 菜单。
2. **无持续的「说话→当前频道 / 收听→…」状态提示**，用户易误以为 PTT 会打到所有已加入群。
3. 服务端用户监听上限拒绝文案误写成 “in this channel”（已在实现中修正）。
4. ini 注释示例 `listenersperuser=2` 对「多群组」业务偏紧，需文档化推荐值。

**结论**：不改语音核心路由；补齐 UX、映射文档、配置说明与小文案修复。

## 2. 产品映射表

| 业务用语 | Mumble 概念 | 实现要点 |
|----------|-------------|----------|
| 群组 | Channel | `channel id` |
| 进入群组 / 当前群组 | 物理成员 `User::cChannel` | `joinChannel` / `UserState.channel_id` |
| 加入多群组（可同时多个） | Channel Listener | `UserState.listening_channel_add` |
| 退出多群组中的某一群 | 取消监听 | `listening_channel_remove` |
| 在群组中听到他人 PTT | 同频道成员 **或** 监听者 | `processMsg` |
| 本人 PTT | **仅**当前 `cChannel`（+链接频道 Speak 等既有规则） | 不遍历本人 listener 集合作为发送目标 |

### 操作规则

1. **切换主频道**（进入另一群）：只改变 `cChannel`；**默认不清理**已有监听列表（保证「人不在该群也能听」）。
2. **退出监听某群**：`stopListeningToChannel`；若该群是当前频道，人仍在该频道，仍能听到同频道语音（成员路径），只是去掉 Listener 代理显示。
3. **退出当前群并离开服务器树中的位置**：需先 `join` 到其它频道；不能「不属于任何频道」。
4. **权限**：目标频道需 `Listen`；默认 ACL 含 Listen，运维可按频道收紧。
5. **批量加入**：客户端 `ServerHandler::startListeningToChannels` 一次发送多个 id；Ice 侧有 `startListening` 等 API。

### 与 Channel Link 的边界

- **Listener**：用户级，适合「多群组收听」。
- **Link**：频道级双向耦合，易导致整树串音；业务多群组**不要**用 Link 代替 Listener。
